# MetricsService

WebAPI для работы с timescale-данными результатов обработки: загрузка CSV, расчёт
интегральных метрик, выборка результатов по фильтрам.

Тестовое задание по направлению «Разработчик C#» (ИнфоТеКС).

## Стек

| Слой | Технологии |
|---|---|
| API | .NET 10, ASP.NET Core, Swagger (Swashbuckle) |
| Доступ к данным | EF Core 10, Npgsql, PostgreSQL 16 |
| Парсинг | CsvHelper |
| Тесты | NUnit, NSubstitute, EF Core In-Memory SQLite |
| SPA | Angular 21 (standalone, signals, zoneless) |
| Инфраструктура | Docker, docker compose, nginx |

Требование ТЗ — .NET 8+; проект собран на .NET 10.

## Быстрый старт

```bash
docker compose up --build
```

| Что | Где |
|---|---|
| SPA | http://localhost:4200 |
| Swagger | http://localhost:8080/swagger |
| PostgreSQL | localhost:5432 (admin / password / metric_db) |

Миграции накатываются автоматически при старте API, база поднимается в контейнере —
дополнительных действий не требуется.

Примеры файлов для проверки лежат в `samples/`: один валидный, три с разными видами
ошибок и пять с разнесёнными характеристиками для проверки фильтров.

### Запуск без Docker

```bash
# 1. Поднять только БД
docker compose up postgres -d

# 2. API (строка подключения — в appsettings.json)
dotnet run --project MetricsService.Api

# 3. SPA (dev-сервер проксирует /api на localhost:5297)
cd MetricsService.Spa && npm install && npm start
```

### Тесты

```bash
dotnet test
```

## API

Базовый маршрут — `api/metrics`.

### 1. Загрузка CSV

```
POST /api/metrics/files
Content-Type: multipart/form-data, поле csvFile
```

Файл парсится, проходит валидацию, значения сохраняются в таблицу `Values`,
рассчитанные интегральные метрики — в `Results`.

Формат строки: `Date;ExecutionTime;Value`, где дата — `ГГГГ-ММ-ДДTчч-мм-сс.ммммZ`.

Правила валидации:

- дата не раньше 01.01.2000 и не позже текущего момента;
- время выполнения ≥ 0;
- значение показателя ≥ 0;
- от 1 до 10 000 строк;
- все три значения присутствуют и соответствуют своим типам.

При нарушении любого условия файл считается невалидным целиком: до записи в БД
дело не доходит, клиент получает `400` с `ValidationProblemDetails`, где перечислены
**все** найденные ошибки с номерами строк, а не только первая.

Повторная загрузка файла с тем же именем перезаписывает его данные.

### 2. Результаты по фильтрам

```
GET /api/metrics/results?FileName=&FirstStartTimeFrom=&FirstStartTimeTo=
    &ValueMeanFrom=&ValueMeanTo=&AvgDurationSecondsFrom=&AvgDurationSecondsTo=
```

Все фильтры опциональны и комбинируются по «И». Условия добавляются к `IQueryable`
только для заданных параметров, поэтому в SQL уходят ровно те предикаты, которые
задал клиент.

### 3. Последние значения файла

```
GET /api/metrics/values/{fileName}/latest
```

Последние 10 записей файла, отсортированных по `Date` по убыванию. Если файла нет —
`404`.

Длительности во всех ответах отдаются числом секунд, а не `TimeSpan`: так удобнее
клиенту и не приходится разбирать формат `00:01:23.4500000`.

## Архитектура

```
MetricsService.Api          контроллеры, DTO запросов/ответов, middleware
MetricsService.Application  парсинг, валидация, расчёт метрик, оркестрация
MetricsService.Domain       модели, DTO обмена между слоями, интерфейсы
MetricsService.Repository   DbContext, конфигурации сущностей, миграции, репозиторий
MetricsService.Test         юнит- и интеграционные тесты
MetricsService.Spa          Angular-приложение
```

Зависимости направлены к домену: `Api → Application → Domain ← Repository`.
Доменные модели не знают про EF — ключи, индексы и ограничения заданы через
`IEntityTypeConfiguration` в слое доступа к данным.

Обработка файла разложена на три независимых шага, каждый из которых тестируется
отдельно: `ICsvParser` (текст → строки + ошибки формата), `ICsvValidator`
(бизнес-правила ТЗ), `IFileProcessingService` (оркестрация и расчёт метрик).

## Принятые решения

**Result вместо исключений для валидации.** Невалидный файл — штатный результат
работы метода, а не исключительная ситуация, поэтому `ProcessAsync` возвращает
`ProcessingResult` со списком ошибок. Сигнатура метода не врёт, ошибки собираются
в один ответ, тесты обходятся без `Assert.Throws`. Непредвиденные исключения ловит
`ExceptionHandlingMiddleware` и отдаёт `500` с `ProblemDetails`, не раскрывая
внутренностей.

**Валидация до обращения к БД.** Требование «откатить изменения» выполняется
структурно: 10 000 строк — небольшой объём, файл целиком парсится в память,
проверяется и только затем пишется. Транзакция нужна лишь для атомарности пары
«перезаписать Values + обновить Result».

**Перезапись через delete + insert.** Состав строк файла меняется целиком, мержить
их построчно бессмысленно. Удаление выполняется через `ExecuteDeleteAsync` — прямой
`DELETE` без загрузки сущностей в память. Поскольку эта операция идёт мимо change
tracker'а, в начале метода вызывается `ChangeTracker.Clear()`.

**Индексы под конкретные запросы.** Для фильтров второго метода — одиночные индексы
на `FirstStartTime`, `ValueMean`, `AvgDuration` (фильтры независимы, составной индекс
помогал бы только одной комбинации, замедляя вставку). Для третьего метода —
составной `(FileName, StartTime)`: PostgreSQL берёт по нему последние 10 записей без
отдельной сортировки. Все читающие запросы выполняются с `AsNoTracking`.

**Даты только в UTC.** Значения приводятся к UTC на границе системы: при парсинге
файла и при разборе параметров фильтра. Внутри системы инвариант «любой DateTime —
это UTC», что снимает проблему Npgsql с `timestamp with time zone`.

**Один origin для SPA и API.** В контейнере nginx раздаёт статику и проксирует
`/api` на бэкенд, в разработке ту же роль играет прокси dev-сервера Angular.
Благодаря этому CORS не нужен ни в одном из окружений.

## Тесты

| Что покрыто | Как |
|---|---|
| Парсер | форматы даты и чисел, пропущенные и лишние поля, накопление ошибок, границы 10 000 строк |
| Валидатор | каждое правило ТЗ, включая границу 01.01.2000 и номера строк в сообщениях |
| Сервис обработки | объединение ошибок парсера и валидатора, отсутствие записи в БД при ошибках, расчёт всех метрик, медиана для чётного количества строк |
| Репозиторий | перезапись по имени файла, изоляция данных других файлов, все четыре фильтра, сортировка и лимит третьего метода |

Тесты репозитория используют SQLite in-memory: они быстрые и не требуют внешних
зависимостей, но проверяют логику на другом провайдере, чем прод. Полностью честную
проверку `ExecuteDeleteAsync` и транзакций на PostgreSQL дали бы интеграционные
тесты на Testcontainers — см. ниже.

## Что можно развить

Осознанно не реализовано, чтобы не выходить за рамки задания:

- **Хранение исходных файлов в объектном хранилище (MinIO / S3).** Сейчас
  оригиналы не сохраняются: по ТЗ они не нужны, имя файла используется только как
  ключ для перезаписи и фильтрации. Складывать исходники в MinIO имело бы смысл,
  если понадобится повторно пересчитать метрики при изменении логики агрегатов или
  доказать происхождение данных. Инфраструктурно это дёшево (ещё один сервис в
  compose и клиент в слое доступа к данным), но добавляет точку отказа ради
  сценария, которого в требованиях нет.
- **Интеграционные тесты на Testcontainers** — проверка репозитория на настоящем
  PostgreSQL вместо SQLite.
- **Пагинация и сортировка** во втором методе: сейчас возвращается вся выборка,
  при росте числа файлов понадобится постраничный вывод.
- **Потоковый парсинг** для файлов, существенно превышающих лимит в 10 000 строк:
  текущая реализация держит файл в памяти, что при заданном лимите оправдано.
- **Аутентификация и закрытие Swagger** для непубличных окружений — в задании
  документация нужна открытой.
